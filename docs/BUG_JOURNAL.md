# Bug Journal — BenchmarkWatcher

## Patterns to scan for FIRST

- **Mobile won't bundle · "Tailwind CSS has not been configured with the NativeWind preset"** → `mobile/tailwind.config.js` MUST include `presets: [require("nativewind/preset")]` (NativeWind v4). Correct `babel.config.js` + `metro.config.js` + `global.css` is **not** enough. Tests/typecheck pass without it; only a real bundle/render catches it.
- **pytest "No module named pytest"** → run `venv/bin/python -m pytest tests`, NOT the Homebrew `python3` (3.14) which has no pytest. A false-failure trap — the suite is actually green (68 passed).
- **Mobile shows "Server error (500)" but the same URL curls 200 locally** → the app reads `EXPO_PUBLIC_API_URL` from `mobile/.env` (prod = `https://benchmarkwatcher.online`) and ignores localhost. To test against local Flask, point `mobile/.env` at `http://localhost:5002`, rebuild with `--clear`, then revert `.env`. An inline `EXPO_PUBLIC_API_URL=...` does NOT win over the `.env` file.
- **zsh does not word-split unquoted `$VAR`** → `perl ... $FILES` passes the whole newline-joined list as ONE filename ("Can't open …: No such file"). Use `find … -exec perl -pi -e '…' {} +` for codemods.
- **Web styling missing locally** → `app/static/css/tailwind.css` is committed empty (0 bytes) and built by `npm run build:css` (CI/deploy). Rebuild after editing tokens in `tailwind.web.config.js` / `base.html`.
- **Alternate view modes silently broken** → a passing DEFAULT view does NOT mean alternate modes work. The grid has 3 card styles (Full / Minimal / Dense) × 2 views (grid / compact) × 7 themes, all client-rendered by imperative DOM surgery in `grid_view.js` / `compact_table.js` that depends on hook classes (`bw-grid-category-row`, `bw-grid-change-pct`, …). A `className` reassignment in ONE reset path silently breaks the others. ALWAYS screenshot every card style + view (+ a dark theme), not just the default.
- **Flask template edits need a server restart to show** → the dev server runs debug-off (no Jinja auto-reload). Static JS/CSS re-fetch on a browser reload, but `.html` template changes only appear after restarting `run.py`. Verify with `curl` against the server (definitive), not the cache-prone preview browser.
- **A fix is live on the server but the browser still shows the bug** → check BOTH: (1) a CLIENT re-render pass (`compact_table.js applyVisualSettings`, `grid_view.js updateGridSettings`) reads a raw `data-*` attribute and overwrites the server-correct value — fix the re-render path, not just the template/initial render; (2) no static cache-busting → the browser runs stale JS. ALWAYS verify by reading the **rendered DOM** (`querySelectorAll` → check `textContent`), never `curl`/unit-tests alone. Static URLs now carry `?v=<mtime>` (see `app/__init__.py _versioned_url_for`). And the float-noise class hides per-FIELD and per-PATH: rounding the absolute change did NOT round the percent; rounding the initial render did NOT cover the range-refresh re-render. Guard it with `tests/e2e/no-raw-floats.spec.ts` (scans the rendered DOM for `/\d+\.\d{5,}/` after each client re-render).
- **A display SETTING works on first load but reverts after a refresh/re-render** → two renderers for the same component (server Jinja template + client JS re-render) drifted out of structural sync, so shared post-render logic (`compact_table.js applyVisualSettings`) silently breaks on the path it wasn't written for. Concretely: the server put `data-value` ON `.chg-cell`/`.pct-cell`; the JS re-render made those WRAPPERS and moved `data-value` to an inner div → `parseFloat(cell.dataset.value)` = NaN → handler bailed. Fix by resolving the value element where it actually lives (`el.closest('[data-value]')`), not by assuming a fixed shape. Better: keep the two renderers' DOM identical. Diff the server HTML vs the JS-rendered HTML when a setting "won't stick".
- **ALL e2e tests fail at once on valid selectors ("element not found")** → before suspecting a code regression, open the Playwright failure **page snapshot** (`test-results/*/error-context.md`). If it shows a *different app*, the cause is `playwright.config.ts reuseExistingServer: true` (local runs) reusing a FOREIGN server already listening on the e2e port. Re-run on a free port: `PLAYWRIGHT_PORT=<free>`. On this machine the default flaskCommand's bare `python` also lacks Flask, so also set `PLAYWRIGHT_FLASK_COMMAND="venv/bin/python -m flask --app run:app run --host 127.0.0.1 --port <port>"`. Default e2e port is now 5781 (was the collision-prone 5050).
- **UI looks blank/unrendered in the Claude Preview but works elsewhere** → the preview tab runs BACKGROUNDED (`document.hidden === true`, `visibilityState === 'hidden'`), so `requestAnimationFrame` is **throttled and never fires**. Any rAF-gated rendering won't run there — e.g. `compact_table.js initSparklines` wraps its whole render in `requestAnimationFrame`, so the compact sparklines show blank in the preview yet render fine in a real foreground browser (Chart.js's first paint is synchronous, which is why the *detail* chart still appears). Don't conclude "broken" from a hidden-preview screenshot/eval — check `document.hidden`, await a rAF, or verify via e2e (Playwright renders).
- **Namecheap "host 500" on EVERY route (the namecheap-branded page, not the app's own error page)** → Passenger isn't booting the app; its virtualenv is broken/incomplete. SSH in and `tail ~/<app>/stderr.log` for the real traceback (usually `ModuleNotFoundError`). Critically, the venv Passenger boots is the one named in the app's `.htaccess` (`PassengerPython "/home/<user>/virtualenv/<app>/<ver>/bin/python"`) — **install deps into THAT venv** (`<that-python> -m pip install -r requirements.txt`), **NOT** the repo's local `venv/` (they're different, and the local one is usually stale too). Then `touch tmp/restart.txt` and confirm `curl -sI https://<host>/ | head -1` → `HTTP/2 200`. The two-venv confusion is the trap — always read `.htaccess` for the real path.
- **A flat / 0% change renders up-coloured (teal ▲) instead of neutral** → market colour is decided binary (`change >= 0 ? up : down`), so an *unchanged* value falls into "up" — contradicting the Market-Pulse breadth, which counts flat correctly. Fix = a `--color-flat` token + 3-way (`>0 up / <0 down / ==0 flat`). The trap: the compact table colours in ~6 JS spots — the initial render AND `applyVisualSettings` (which RE-COLOURS on load: sign, chg colour, pct sign, pct-badge colour+bg) — PLUS the server template PLUS the grid renderers. Fix every spot together or `applyVisualSettings` silently re-overrides the rest (classic renderer drift). `data-direction` in `price_card.html` already computes 3-way — reuse it.

### 2026-06-07 · PROD outage RESOLVED — Passenger venv missing flask_caching (host 500)

### 2026-06-07 · Grid cards lost up/down colour on the % change (every theme)
- Symptom: the grid card `%` change rendered as plain text — dark in light/dark, amber in the Bloomberg theme — instead of teal-up / claret-down. I first screenshot-audited it as "teal/claret, fine".
- Cause: `updateSettings` wipes EVERY card child's inline style up-front (`card.querySelectorAll('*').forEach(el => el.style.cssText = '')`, ~line 755), then each card-style handler is supposed to re-apply what it needs. The re-apply was incomplete: the **Full Card** % handler blanked it again (`cssText = ''`), and the direction-badge colour was re-applied by NO handler (Full Card didn't, Dense set only font/padding). So both the `%` and the ▲/▼ badge inherited plain text — dark in light/dark, amber in Bloomberg.
- Fix: re-apply `color/background/border: var(--color-up|--color-down...)` from `link.dataset.changePct` in the Full Card handler (% + badge) and the Dense handler (badge). Minimal hides the category row, so its badge is moot. Verified via `getComputedStyle`: teal `rgb(13,118,128)` / claret `rgb(153,15,61)` for % AND badge, in Full Card + Dense, light + Bloomberg.
- Lesson: **verify COLOUR with `getComputedStyle`, not a screenshot eyeball** — I misjudged dark-as-teal and amber-as-up twice this session; the computed value is the only truth (same spirit as "read the rendered DOM, not curl"). And a `style.cssText = ''` reset silently drops template-set inline styling — re-apply, never blank.

### 2026-06-07 · Compact display settings silently stopped applying after a range refresh
- Symptom: change format/colour + percent style/decimals worked on first load but quietly reverted after clicking a range button in compact view. The raw-percent float (below) was the visible tip of the same root cause.
- Cause: structural drift between the two renderers. The SERVER template puts `data-value` ON `.chg-cell`/`.pct-cell`; the JS re-render makes those positioning WRAPPERS and moves `data-value` to an inner div (+ a tooltip sibling). `applyVisualSettings` read `data-value` off the wrapper → NaN → bailed (pct) or mis-targeted the colour (chg, then overridden by the inner inline colour).
- Fix: resolve the value element from wherever it lives — `el.closest('[data-value]')` for `.chg-value`, `cell.matches('[data-value]') ? cell : cell.querySelector('[data-value]')` for `.chg-cell`/`.pct-cell` — and apply text/colour to that element. `| round(2)` added to the server pct as defence-in-depth. Unit test `compactTableApplyVisualSettings.test.js` pins both shapes; jest 151 / pytest 69 / e2e 14.
- Lesson: see the new pattern above — two renderers for one component must emit identical DOM, or shared post-render JS breaks on the untested path.

### 2026-06-07 · Compact-table PERCENT rendered raw floats after a range refresh
- Symptom: clicking a range button in compact view re-rendered the % column with full float precision (`+5.621805792163545%`, `+87.48500599760096%`). Initial server render was clean; the client re-render was not. The new e2e regression guard caught it.
- Cause: `compact_table.js` rounds the absolute change (`roundedChange`, toFixed 4) but rendered `displayChangePercent` RAW (~line 208). `applyVisualSettings` can't compensate — it reads `data-value` off `.pct-cell` (the OUTER div, where the attr is absent) → NaN → early-returns, so the innerHTML render is the only thing shown.
- Fix: add `roundedPercent` (toFixed 2, mirrors roundedChange) used by the cell + tooltip + data-value. Added `tests/e2e/no-raw-floats.spec.ts` (compact + grid + leaderboard, scanned after the client re-render).
- Lesson: the earlier "raw floats" fix covered the absolute change only — the float-noise class hides per-FIELD and per-re-render-PATH. Scan every numeric field in the rendered DOM after each client re-render, not just the default view.
